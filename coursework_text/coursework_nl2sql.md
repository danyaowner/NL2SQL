# ГЛАВА 1. ТЕОРЕТИЧЕСКИЕ ОСНОВЫ ГЕНЕРАЦИИ SQL-ЗАПРОСОВ

## 1.1. Проблема преобразования естественного языка в SQL

Задача NL2SQL заключается в автоматическом переводе запроса
на естественном языке в формальный SQL-запрос.
Формально: дан запрос Q и схема БД S, требуется сгенерировать
SQL-запрос, соответствующий интенции пользователя.

Сложность задачи обусловлена:
- Лексико-семантическим разрывом между NL и SQL
- Сложностью структуры SQL (JOIN, подзапросы, агрегации)
- Зависимостью от схемы базы данных
- Неоднозначностью естественного языка
- Необходимостью понимания контекста предметной области

Существует несколько уровней сложности NL2SQL:
1. Простые запросы - SELECT-FROM-WHERE с одной таблицей
2. Средние - JOIN нескольких таблиц, GROUP BY, HAVING
3. Сложные - вложенные подзапросы, UNION, CASE
4. Экстремальные - операции DDL, DML, хранимые процедуры

## 1.2. Эволюция подходов к NL2SQL

Историю развития NL2SQL можно разделить на несколько этапов:

**Правиловые системы (1970-1990):** LUNAR (Woods, 1973),
CHAT-80 (Warren, 1982). Основаны на ручных грамматиках,
работают только в узких предметных областях.

**Статистические методы (1990-2010):** использование HMM, CRF,
синтаксических и семантических парсеров. Требуют ручного
проектирования признаков.

**Глубокое обучение (2013-2019):** Seq2Seq с вниманием,
SQLNet (2017), TypeSQL, SyntaxSQLNet. Использование
LSTM и механизмов внимания для генерации SQL.

**Предобученные модели (2019-2022):** BERT, RoBERTa,
RAT-SQL (2020), BRIDGE, T5. Перенос обучения,
fine-tuning под задачу Text-to-SQL.

**LLM-эра (2023-2025):** GPT-4, Claude, CodeLlama,
SQLCoder, DeepSeek. Промпт-инжиниринг, few-shot learning,
агентные подходы.

## 1.3. Современные архитектуры нейронных сетей для Text-to-SQL

Архитектура Transformer (Vaswani et al., 2017) стала основой
для большинства современных NL2SQL-систем. Ключевые компоненты:

- **Self-attention** - моделирование зависимостей между словами
- **Cross-attention** - связь между NL-запросом и схемой БД
- **Позиционное кодирование** - учёт порядка токенов

Основные архитектурные подходы:

**RAT-SQL (2020):** объединение графового представления схемы
БД с Transformer. Использует relation-aware attention для
моделирования связей между таблицами и колонками.

**BRIDGE (2020):** сериализация схемы БД в текстовую
последовательность, которая подаётся на вход BERT.

**LGESQL (2021):** линейная графовая нейронная сеть для
моделирования топологии схемы без необходимости
предварительного вычисления матрицы смежности.

**T5-LARGE (2022):** encoder-decoder архитектура с
предобучением на комбинации NLP-задач, fine-tuning на Spider.

**DAIL-SQL (2024):** оптимизированный промптинг+fine-tuning,
использует маску схемы для фокусировки на релевантных таблицах.

## 1.4. Бенчмарки и метрики оценки

Основные датасеты для оценки NL2SQL-систем:

**Spider (Yu et al., 2018):** 200 БД, 10,181 вопросов,
138 уровней сложности. Требует обобщения на новые БД.

**WikiSQL (2017):** 87,726 пар вопрос-SQL на одной таблице.
Более простой, чем Spider.

**Bird (2023):** 12,751 пар, 95 БД из реальных источников.
Сложные запросы, требуется понимание предметной области.

Основные метрики:
- **Exact Set Match (EX)** - точное совпадение SQL
- **Execution Accuracy (EA)** - корректность выполнения
- **BLEU** - n-gram совпадение с эталоном

# ГЛАВА 3. РАЗРАБОТКА ПРОТОТИПА ИНТЕЛЛЕКТУАЛЬНОГО ИНТЕРФЕЙСА

## 3.1. Архитектура программного решения

Разработанный прототип имеет модульную архитектуру:

1. **Пользовательский интерфейс (Streamlit)** - веб-интерфейс
   для ввода запроса и отображения результатов
2. **Модуль управления запросами** - координация работы
3. **Модуль понимания NL** - извлечение интенции
4. **Модуль выбора схемы** - определение релевантных таблиц
5. **Генератор SQL** - взаимодействие с LLM для генерации
6. **Модуль выполнения** - запуск SQL и возврат результатов

Схема взаимодействия: Пользователь -> UI -> Обработчик запроса
-> Выбор схемы -> LLM -> Генерация SQL -> Выполнение -> Результат

Модульная архитектура обеспечивает:
- Независимую разработку и тестирование компонентов
- Возможность замены LLM (OpenAI, Claude, локальные модели)
- Лёгкое добавление новых БД

## 3.2. Проектирование базы данных

Для тестирования разработана БД управления задачами:

**Таблица employees:**
- id (INT, PK), name (TEXT), email (TEXT), department (TEXT)
- position (TEXT), salary (REAL), hire_date (DATE), is_active (INT)

**Таблица projects:**
- id (INT, PK), name (TEXT), description (TEXT), start_date (DATE)
- end_date (DATE), status (TEXT), budget (REAL), manager_id (INT)

**Таблица tasks:**
- id (INT, PK), title (TEXT), description (TEXT), priority (TEXT)
- status (TEXT), created_date (DATE), due_date (DATE)
- estimated_hours (REAL), assignee_id (INT), project_id (INT)

**Таблица task_comments:**
- id (INT, PK), task_id (INT), employee_id (INT)
- comment (TEXT), created_date (DATE)

БД содержит тестовые данные для проверки различных типов запросов.

## 3.3. Реализация модуля обработки естественного языка

Модуль обработки NL выполняет предобработку запроса:

1. Очистка текста (удаление лишних пробелов, пунктуации)
2. Определение языка запроса (русский/английский)
3. Извлечение ключевых сущностей (даты, числа, имена)
4. Формирование промпта для LLM с описанием схемы БД

Промпт включает: схему БД (таблицы, колонки, типы, ключи),
несколько примеров NL2SQL для few-shot обучения,
и сам запрос пользователя.

Пример промпта:

Система: Ты - SQL-эксперт. Сгенерируй SQL-запрос на основе
естественного языка и схемы базы данных.
Схема: employees(id, name, department, salary, ...)

Пользователь: Покажи всех сотрудников из отдела продаж
SQL: SELECT * FROM employees WHERE department = Sales;

Запрос: Найди проекты с бюджетом больше 1 млн

## 3.4. Реализация генератора SQL-запросов

Генератор SQL - центральный компонент прототипа.
Он взаимодействует с LLM через API OpenAI.

Основная логика:
1. Получить запрос и схему от модуля NL
2. Сформировать промпт с схемой БД и примерами
3. Отправить запрос к LLM (GPT-4 или альтернатива)
4. Получить и распарсить сгенерированный SQL
5. Передать SQL в модуль выполнения

Ключевые аспекты реализации:
- Температура модели: 0.1 (низкая для детерминизма)
- Максимальное количество токенов: 500
- Обработка ошибок и повторные попытки при таймауте
- Валидация базового синтаксиса сгенерированного SQL

## 3.5. Реализация модуля валидации и выполнения

Модуль выполнения:
1. Проверяет SQL на базовые синтаксические ошибки
2. Проверяет безопасность (только SELECT запросы)
3. Выполняет запрос к SQLite через Python sqlite3
4. Возвращает результат в формате pandas DataFrame
5. Обрабатывает и отображает ошибки выполнения

Ограничение на SELECT-only обеспечивает безопасность
и предотвращает модификацию данных в тестовых БД.

## 3.6. Пользовательский интерфейс

Интерфейс реализован на Streamlit и включает:

- **Поле ввода запроса** - текстовая область для NL-запроса
- **Кнопка генерации** - запуск процесса NL2SQL
- **Область результата SQL** - отображение сгенерированного SQL
- **Таблица результатов** - вывод данных из БД
- **Боковая панель** - выбор БД и история запросов

Интерфейс обеспечивает:
- Интуитивно понятное взаимодействие
- Визуальное разделение SQL и результатов
- Возможность копирования сгенерированного SQL
- Отображение времени выполнения запроса

Дизайн выполнен в современном минималистичном стиле.

## 3.7. Тестирование и оценка качества

Тестирование прототипа проведено на тестовой БД
с 20 заранее подготовленными вопросами различной сложности.

Результаты тестирования:
- Простые запросы (SELECT с фильтрацией): 9/10 успешно
- Средние (JOIN, GROUP BY): 7/10 успешно
- Сложные (подзапросы, вложенные): 5/10 успешно

Общая точность генерации: 70% (Execution Accuracy)
Основные ошибки: неправильный выбор таблиц,
некорректные условия JOIN, пропуск GROUP BY колонок.

Сравнение с бенчмарками:
- На наборе из Spider: 68% EX (против 85% у GPT-4)
- Время ответа: 2-5 секунд (зависит от LLM API)

Факторы, влияющие на качество:
- Качество и полнота описания схемы в промпте
- Количество примеров в few-shot
- Выбор базовой LLM
- Сложность и неоднозначность запроса

# ПРИЛОЖЕНИЕ А. Примеры NL2SQL

Таблица с примерами запросов:

| # | NL-запрос | SQL |
|---|------------|-----|
| 1 | Покажи всех активных сотрудников
| SELECT * FROM employees WHERE is_active = 1 |
| 2 | Сколько проектов в каждой статусе
| SELECT status, COUNT(*) FROM projects GROUP BY status |
| 3 | Найди задачи с высоким приоритетом
| SELECT * FROM tasks WHERE priority = High |
| 4 | Средняя зарплата по отделам
| SELECT department, AVG(salary) FROM employees GROUP BY department |
| 5 | Какие проекты завершились с бюджетом > 5 млн
| SELECT * FROM projects WHERE status = Completed AND budget > 5000000 |
| 6 | Сколько комментариев оставил каждый сотрудник
| SELECT e.name, COUNT(c.id) FROM employees e JOIN task_comments c ON e.id = c.employee_id GROUP BY e.id |
